# 🌿 Робочий процес (Workflow) та Контекст для ESP32 (MoonPlants)

Цей документ створено як єдине джерело істини (Single Source of Truth) для розуміння того, як фізичний мікроконтролер (ESP32) взаємодіє з бекендом MoonPlants. Документ призначений для AI агентів та розробників, щоб не проводити повторний ресерч.

---

## 🏗 Архітектурний огляд

Система використовує модель **Polling + REST API**. ESP32 виступає як HTTP клієнт, який періодично надсилає дані (телеметрію) та запитує команди на виконання від сервера.

* **Бекенд:** Next.js API Routes (`/app/api/iot/v1/*`)
* **База даних:** Supabase (PostgreSQL)
* **Автентифікація пристроїв:** HMAC-SHA256 (без передачі секрету по мережі)

---

## 🔄 Життєвий цикл пристрою (Device Lifecycle)

### 1. Provisioning (Етап виробництва/прошивки)
Кожен ESP32 повинен отримати 3 унікальних значення, які записуються в його енергонезалежну пам'ять (NVS/EEPROM):
1. `DeviceId` (UUID)
2. `HMAC Secret` (32 байти, для підпису запитів)
3. `Claim Code` (Код для користувача, щоб прив'язати пристрій до акаунта)

*У проекті це імітується скриптом `scripts/create-test-device.ts`, який створює тестовий пристрій в базі Supabase (`devices`, `device_credentials`).*

### 2. Claiming (Прив'язка користувачем)
Пристрій за замовчуванням має статус `unclaimed`. Користувач вводить `Claim Code` в UI MoonPlants. Сервер шукає пристрій, перевіряє хеш коду, і змінює статус на `claimed`, призначаючи `owner_user_id`. Тільки після цього пристрій може повноцінно працювати з рослинами користувача.

### 3. Active Loop (Основний цикл роботи ESP32)
ESP32 працює в нескінченному циклі (або прокидається з Deep Sleep).
* **Синхронізація часу:** Обов'язкова умова! ESP32 має підключитись до NTP сервера, оскільки для HMAC-підпису потрібен точний Unix Timestamp (допуск ±120 сек).
* **Телеметрія:** Збір даних з датчиків і відправка `POST /api/iot/v1/measurements`.
* **Опитування команд:** Запит `GET /api/iot/v1/commands`.
* **Виконання та Підтвердження:** Якщо є команди (напр., увімкнути помпу), ESP32 виконує фізичну дію і надсилає `POST /api/iot/v1/commands/{id}/ack`.

## 🔐 Безпека та HMAC підписи

Кожен запит від ESP32 захищений HMAC-SHA256. 
Обов'язкові HTTP заголовки:
* `X-Device-Id`: UUID пристрою.
* `X-Device-Seq`: Монотонний лічильник. Збільшується з кожним запитом. Зберігається в NVS ESP32. Сервер відхиляє `seq <= last_seq` для захисту від Replay атак.
* `X-Device-Timestamp`: Unix time (сек).
* `X-Content-SHA256`: SHA-256 хеш тіла запиту (для GET - хеш пустого рядка `""`).
* `X-Device-Signature`: Base64URL від HMAC-SHA256 підпису канонічного рядка.

**Канонічний рядок (Canonical String):**
`METHOD\nPATH\nDEVICE_ID\nSEQ\nTIMESTAMP\nCONTENT_HASH`

## 📡 API Ендпоїнти (ESP32 -> Server) 

Всі знаходяться за адресою: `POST/GET /api/iot/v1/*`

1. `POST /measurements` 
   - Відправляє дані: температура, вологість повітря, освітленість (Lux), рівень заряду (Battery V), RSSI, та масив вологості ґрунту по каналах.
   - **Важливо для ґрунту:** ESP32 надсилає **сирі значення АЦП (Raw ADC)**. Калібрування (переведення у відсотки 0-100%) відбувається на стороні сервера. Це дозволяє легко коригувати формули калібрування без перепрошивки пристрою.
2. `GET /commands?limit=N`
   - Отримує чергу команд для виконання: `PUMP_WATER` (полив), `LIGHT_ON` (увімкнути світло), `LIGHT_OFF` (вимкнути).
3. `POST /commands/{id}/ack`
   - **Критичний крок:** ESP32 має підтвердити виконання команди (`status: "ok"` або `"failed"`). Тільки після отримання `ACK` сервер записує подію в історію (наприклад, `watering_events`).
4. `GET /device-config`
   - Отримання конфігурації пристрою (інтервали опитування, налаштування каналів).

## 🛠 Фізична реалізація: Вимоги до коду ESP32 (C/C++)

Перед тим як почати складати фізичну частину контролера, AI агент повинен врахувати наступне при написанні прошивки:

### 1. Датчики вологості ґрунту (Ємнісні)
* Підключення до пінів ADC (Analog to Digital Converter).
* **Фільтрація шумів:** Обов'язково застосовувати медіанний фільтр (напр., зчитувати 10-20 разів з інтервалом в 10мс, відкидати аномалії і брати середнє/медіану) перед відправкою на сервер.

### 2. Керування помпами (PUMP_WATER)
* Команда з сервера приходить з параметром `water_ml` (мілілітри).
* Сервер не знає пропускну здатність конкретної помпи. **ESP32 має конвертувати мілілітри в мілісекунди роботи помпи** (наприклад, помпа качає 10 мл/сек -> для 50 мл треба увімкнути реле на 5 секунд).
* Команда також містить `max_duration_sec` як failsafe (захист від затоплення), якщо розрахунок мілісекунд вийшов занадто великим.

### 3. Керування світлом (LIGHT_ON / LIGHT_OFF)
* `LIGHT_ON` може містити параметр `duration_sec`. Якщо він є, ESP32 вмикає реле фітолампи і запускає внутрішній апаратний або програмний таймер на вимкнення. Якщо параметра немає — світить нескінченно до команди `LIGHT_OFF`.

### 4. Збереження стану (NVS)
* Бібліотека `Preferences.h` або `nvs_flash` для збереження `seq` (монотонного лічильника). Якщо пристрій перезавантажиться і `seq` скинеться до 0, сервер відхилить всі запити (через anti-replay захист). Тому `seq` треба періодично або після кожного запиту записувати в енергонезалежну пам'ять.

### 5. Deep Sleep vs Delay
* Якщо пристрій живиться від акумулятора: між опитуваннями (напр. 5-10 хвилин) ESP32 має переходити в Deep Sleep. Таймери для світла/помп в такому випадку треба опрацьовувати обережно (не засинати, поки увімкнена помпа!).
* Якщо від мережі (БЖ 5V): можна використовувати звичайний delay/millis() loop.

## 🧪 Симуляція та Тестування (Вже реалізовано)

Для перевірки логіки бекенду без фізичного ESP32 створено симулятор на Node.js:

1. Створення тестового девайса в БД:
   ```bash
   npx tsx scripts/create-test-device.ts
   ```
2. Запуск симулятора (безкінечний цикл відправки даних та обробки команд):
   ```bash
   ./run_simulator.ps1
   # або
   npx tsx scripts/esp32-simulator.ts "00000000-0000-0000-0000-000000000001" "test_secret_key_123"
   ```

**Як перевірити полив через симулятор:**
1. Запустити `create-test-device.ts`.
2. В UI (або через Postman) "заклаймити" пристрій (код `TEST-CLAIM-CODE-2026`).
3. Створити рослину, прив'язати її до `Channel 1` цього девайса.
4. Натиснути кнопку "Полити зараз" в UI. У БД (`device_commands`) створиться запис.
5. Запустити `run_simulator.ps1`. Симулятор підхопить команду, зачекає 1 секунду (імітація роботи) і відправить `ACK`.
6. Сервер обробить `ACK` і запише `watering_events`.
