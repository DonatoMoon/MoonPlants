# 🌿 MoonPlants: IoT Device Integration Guide (v2.1)

Цей посібник містить усю необхідну технічну інформацію для розробки прошивки (firmware) мікроконтролера (ESP32 / ESP8266 / STM32), сумісного з платформою MoonPlants.

---

## 1. Концепція підключення

Кожен пристрій повинен мати унікальний **Device ID** (UUID) та **HMAC Secret** (32 байти), які прошиваються на етапі виробництва. 
Користувач прив'язує пристрій до свого акаунта за допомогою **Claim Code**.

- **Base URL:** `http://<your-domain-or-ip>:3000/api/iot/v1`
- **Протокол:** HTTP/1.1 (рекомендовано HTTPS у продакшні)
- **Формат даних:** JSON (UTF-8)

---

## 2. Авторизація (HMAC-SHA256)

Кожен запит від пристрою ПОВИНЕН містити набір заголовків для автентифікації. Якщо підпис невірний, сервер поверне `401 Unauthorized`.

### Обов'язкові заголовки:
- `X-Device-Id`: UUID пристрою.
- `X-Device-Seq`: Монотонний лічильник запитів (uint64). Має збільшуватися з кожним запитом. Сервер відхиляє запити з `seq <= last_stored_seq` (захист від Replay атак).
- `X-Device-Timestamp`: Поточний Unix Timestamp (секунди). Допуск ±120 сек.
- `X-Content-SHA256`: SHA256 хеш тіла запиту (HEX). Для GET-запитів використовуйте хеш порожнього рядка.
- `X-Device-Signature`: HMAC-SHA256 підпис (Base64URL).

### Формування підпису:
1. Створіть канонічний рядок (Canonical String), з'єднавши параметри через символ перенесення рядка `\n`:
   ```
   METHOD\nPATH\nDEVICE_ID\nSEQ\nTIMESTAMP\nCONTENT_HASH
   ```
   *Приклад для POST /api/iot/v1/measurements:*
   ```
   POST\n/api/iot/v1/measurements\n00000000-0000-0000-0000-000000000001\n12345\n1710515000\n9ce884bd1d16c5d27e871afcf0d63878b2679c855878fb6739898b44001bff12
   ```

2. Розрахуйте `HMAC-SHA256` від цього рядка, використовуючи ваш `HMAC Secret`.
3. Результат перетворіть у **Base64URL** (Base64, де `+` замінено на `-`, `/` на `_`, а `=` видалено).

## 3. Основні ендпоїнти

### 3.1 Надсилання телеметрії
**Ендпоїнт:** `POST /measurements`

**Тіло запиту:**
```json
{
  "measuredAt": 1710515000,
  "air": { "tempC": 24.5, "humidityPct": 45.0 },
  "lightLux": 850,
  "soil": [
    { "channel": 1, "moistureRaw": 450 },
    { "channel": 2, "moistureRaw": 610 }
  ],
  "batteryV": 3.85,
  "rssiDbm": -65
}
```
*Примітка:* `moistureRaw` — це сире значення з АЦП. Сервер сам перетворить його у % за формулою калібрування.

---

### 3.2 Отримання команд
Пристрій повинен періодично (напр. раз на 1 хв) опитувати сервер на наявність команд (полив, світло).

**Ендпоїнт:** `GET /commands?limit=5`

**Відповідь:**
```json
{
  "commands": [
    {
      "id": "uuid-123",
      "type": "PUMP_WATER",
      "payload": { "channel": 1, "water_ml": 150, "max_duration_sec": 30 }
    },
    {
      "id": "uuid-456",
      "type": "LIGHT_ON",
      "payload": { "duration_sec": 3600 }
    },
    {
      "id": "uuid-789",
      "type": "LIGHT_OFF",
      "payload": {}
    }
  ]
}
```

#### Деталі виконання команд:
1. **`PUMP_WATER`**: Команда містить `water_ml` (об'єм у мілілітрах). Оскільки серверам невідома точна пропускна здатність конкретної помпи, **мікроконтролер має сам перерахувати мілілітри в мілісекунди роботи помпи** (напр., якщо помпа дає 10 мл/сек, то 150 мл = 15 секунд роботи). Сервер також передає `max_duration_sec` як запобіжник (failsafe), щоб уникнути затоплення в разі помилки розрахунків.
2. **`LIGHT_ON`**: Якщо передано `duration_sec`, пристрій має увімкнути фітолампу і запустити внутрішній таймер на вимкнення. Якщо `duration_sec` не передано, лампа вмикається на невизначений час (до команди `LIGHT_OFF`).
3. **`LIGHT_OFF`**: Негайне вимкнення фітолампи.

---

### 3.3 Підтвердження виконання (ACK)
Після виконання команди пристрій ОБОВ'ЯЗКОВО має надіслати підтвердження. Без цього статус команди залишиться "sent", а полив не буде зараховано в історію.

**Ендпоїнт:** `POST /commands/{commandId}/ack`

**Тіло запиту:**
```json
{
  "status": "ok", 
  "executedAt": 1710515100,
  "result": { "durationSec": 15, "waterMlDelivered": 150 }
}
```
*Параметр `status` може бути `ok`, `failed` або `partial`.*

---

### 3.4 Отримання конфігурації
Використовується при старті пристрою для синхронізації налаштувань.

**Ендпоїнт:** `GET /device-config`

**Відповідь:** містить статус `claimed`, назву пристрою та прив'язку рослин до каналів.

## 4. Калібрування датчиків вологості (Server-side vs Edge)

**Чому ми надсилаємо сирі дані (Raw) на сервер, а не відсотки?**
В індустріальному IoT калібрування (перетворення Raw у %) найкраще виконувати на боці сервера. Це пов'язано з тим, що:
1. Датчики вологості деградують з часом, і їхні "сухі/мокрі" межі зміщуються.
2. Різні типи ґрунту (пісок vs торф) мають різний базовий опір/ємність.
3. Оновити формулу калібрування в базі даних (на сервері) для конкретного користувача набагато легше і безпечніше, ніж випускати OTA-оновлення прошивки ESP32.

Сервер MoonPlants використовує лінійну апроксимацію:
- `800` (Raw АЦП) = `0%` (Повністю сухий датчик у повітрі)
- `300` (Raw АЦП) = `100%` (Датчик у склянці з водою)

**Тому мікроконтролер має просто зчитувати АЦП (бажано із застосуванням медіанного фільтра для згладжування шумів) і відправляти ці сирі значення як `moistureRaw`.**

## 5. Алгоритм роботи прошивки (Firmware Flow)

1. **Setup**: Зчитати `DeviceId` та `Secret` з енергонезалежної пам'яті (NVS/EEPROM).
2. **Loop**:
   - Зчитати сенсори (Air, Light, Soil).
   - Розрахувати HMAC підпис та надіслати `POST /measurements`.
   - Надіслати `GET /commands`.
   - Якщо в масиві `commands` є завдання:
     - Для кожної команди:
       - Виконати фізичну дію (ввімкнути помпу/реле).
       - Розрахувати підпис та надіслати `POST /commands/{id}/ack`.
   - Перейти в режим Deep Sleep на `N` секунд або чекати таймера.

## 6. Приклад розрахунку Signature (Pseudo-C++)

```cpp
String deviceId = "0000-0000-...";
String secret = "your_secret";
uint64_t seq = 12345;
uint32_t ts = 1710515000;
String body = "{\"measuredAt\":1710515000,...}";

// 1. SHA256 of body
String contentHash = sha256_hex(body);

// 2. Canonical string
String canonical = String("POST\n/api/iot/v1/measurements\n") + 
                   deviceId + "\n" + 
                   String(seq) + "\n" + 
                   String(ts) + "\n" + 
                   contentHash;

// 3. HMAC-SHA256
byte hmacResult[32];
mbedtls_md_hmac(mbedtls_md_info_from_type(MBEDTLS_MD_SHA256), 
                (byte*)secret.c_str(), secret.length(), 
                (byte*)canonical.c_str(), canonical.length(), 
                hmacResult);

// 4. Base64URL
String signature = base64url_encode(hmacResult, 32);
```
